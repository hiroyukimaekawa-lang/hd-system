# 食べログスクレイピング高速化 修正要件

## 1. 目的

現在の食べログスクレイピングは安全寄りの設定になっており、取得速度が遅い。

現状は、食べログの初期設定が以下になっている。

```text
並列数：2
待機時間：2000ms
リトライ回数：1
タイムアウト：20秒
最大ページ数：20
```

これを、失敗率を抑えつつ高速化したい。

---

## 2. 現状の主な機能

現在の拡張機能には、以下の機能がある。

```text
食べログ検索結果ページの取得
食べログ店舗詳細ページの取得
ページネーション対応
人気ジャンル一括取得
HD対象ジャンルへの正規化
住所取得
電話番号取得
営業時間取得
定休日取得
HP有無判定
CSV出力
失敗URLログ
ブロック検知
自動減速
リトライ
タイムアウト
詳細カウンター表示
```

---

## 3. 現在遅い理由

### 3-1. 詳細ページ取得前に毎回待機している

現在は、店舗詳細ページを取得する前に、毎回 `tabelogDelay` で待機している。

初期値が `2000ms` のため、1件ごとに2秒近い待機が発生する。

例：

```text
20件 ÷ 並列2 × 2秒 = 最低20秒
```

実際には通信時間・HTML解析・リトライもあるため、さらに時間がかかる。

---

### 3-2. 食べログの並列数が最大3に制限されている

現在のUIバリデーションでは、食べログの並列数は最大3になっている（`src/popup.js`）。

```javascript
if (tc > 3) tc = 3;
```

安全ではあるが、安定している環境では遅い。

---

### 3-3. 待機時間が最小1000msに制限されている

現在のUIバリデーションでは、食べログ待機時間の最小値は1000msになっている（`src/popup.js`）。

```javascript
if (td < 1000) td = 1000;
```

これも安全だが、安定環境ではやや遅い。

---

### 3-4. 人気ジャンル一括取得がジャンル単位で直列処理

現在は、人気ジャンルを1つずつ順番に処理している（`src/offscreen.js`）。

```text
カフェ
↓
居酒屋
↓
ラーメン
↓
焼肉
```

さらにジャンル間で待機が入っているため、全ジャンル取得では時間がかかる。

---

### 3-5. ジャンルをまたいだ重複URLの再取得が起きる可能性がある

同じ店舗が複数ジャンルに出る場合、人気ジャンルごとに再取得される可能性がある。

例：

```text
カフェにも出る
スイーツにも出る
パン屋にも出る
```

この場合、同じ店舗詳細ページを複数回fetchしてしまう。

---

## 4. まず行う設定変更

初期設定を以下に変更する。

```text
食べログ 並列数：3
食べログ 待機時間：1000ms
リトライ回数：1
タイムアウト：15秒
最大ページ数：20
```

速度重視モードでは以下。

```text
食べログ 並列数：4
食べログ 待機時間：700ms
リトライ回数：0〜1
タイムアウト：10〜12秒
```

安全重視モードでは以下。

```text
食べログ 並列数：2
食べログ 待機時間：2000ms
リトライ回数：1
タイムアウト：20秒
```

## 5. UI修正要件

### 5-1. 食べログ並列数の上限変更

現在（`src/popup.js`）：

```javascript
if (tc > 3) tc = 3;
```

修正後：

```javascript
if (tc > 5) tc = 5;
```

popup.html側の該当input（`tabelogConcurrency`）の `max="3"` も `max="5"` に合わせて変更する。

---

### 5-2. 食べログ待機時間の下限変更

現在（`src/popup.js`）：

```javascript
if (td < 1000) td = 1000;
```

修正後：

```javascript
if (td < 500) td = 500;
```

popup.html側の該当input（`tabelogDelay`）の `min="1000"` も `min="500"` に合わせて変更する。

---

### 5-3. 速度モードを追加

UIに以下の選択肢を追加する。

```text
安全モード
標準モード
高速モード
```

各モードの設定：

```text
安全モード：
食べログ 並列数 2
待機時間 2000ms
リトライ 1
タイムアウト 20秒

標準モード：
食べログ 並列数 3
待機時間 1000ms
リトライ 1
タイムアウト 15秒

高速モード：
食べログ 並列数 4
待機時間 700ms
リトライ 0
タイムアウト 10秒
```

モード選択時に、`tabelogConcurrency` / `tabelogDelay` / `maxRetries` / `fetchTimeout` の各inputへ自動反映する（既存の「高度な設定」欄の値を上書きする形でよい）。

## 6. オフスクリーン処理の修正要件（`src/offscreen.js`）

### 6-1. 一覧URL収集フェーズと詳細取得フェーズを分ける

現在は、ページごとに以下の流れになっている。

```text
一覧ページ取得
↓
店舗詳細取得
↓
次ページ取得
↓
店舗詳細取得
```

修正後は以下にする。

```text
一覧ページをページネーション終了まで取得
↓
店舗URLを全件収集
↓
URL重複排除
↓
詳細ページを並列取得
↓
CSV出力
```

追加する関数：

```javascript
async function collectTabelogStoreUrls(listUrl, options) {
  // maxPagesまで一覧ページを回り、店舗URLだけを集める
}
```

```javascript
async function fetchTabelogDetailsBatch(storeUrls, options) {
  // 店舗詳細URLを並列取得する
}
```

既存のブロック検知・自動減速・カウンター更新（成功/失敗/リトライ中/ブロック/タイムアウト等）は、`fetchTabelogDetailsBatch` 内に引き継ぐこと。

---

### 6-2. 人気ジャンル間の重複URLをスキップする

人気ジャンル一括取得（1696行目付近のループ）では、親タスクに `globalSeenUrls` を持たせる。

```javascript
const globalSeenUrls = new Set();
```

各ジャンルで店舗URLを取得した後、すでに取得済みのURLは除外する。

```javascript
urls = urls.filter(url => {
  const normalized = normalizeListUrl(url);
  if (globalSeenUrls.has(normalized)) return false;
  globalSeenUrls.add(normalized);
  return true;
});
```

`normalizeListUrl` が未実装の場合は、クエリパラメータ・末尾スラッシュの違いを吸収する正規化関数として新規追加する。

---

### 6-3. ジャンル間待機を短縮

現在、1762行目付近で以下のように待機している。

```javascript
if (pt && pt.running) await sleep(2000);
```

修正後：

```javascript
if (pt && pt.running) await sleep(500);
```

ただし、アクセス制限を検知した場合は、以下にする。

```javascript
if (pt && pt.running) await sleep(5000);
```

---

### 6-4. 自動減速は維持する

403 / 429 / ブロック画面を検知した場合は、現在と同様に減速する。

```text
並列数：1
待機時間：7000ms以上
```

これは残す。既存のブロック検知処理・カウンター（`cntBlock`等）の呼び出し箇所を変更しないこと。

---

### 6-5. 失敗率に応じた自動調整

以下のルールを追加する。

```text
直近20件で失敗率20%以上
→ 並列数を1下げる
→ 待機時間を+1000ms

直近20件で失敗率5%以下
→ 並列数を1上げる
→ 待機時間を-300ms
```

上限・下限：

```text
並列数：1〜5
待機時間：500〜7000ms
```

---

## 7. 詳細ページ取得の最適化

### 7-1. 必要なセレクタだけを見る

詳細ページ解析では、必要な要素だけを取得する。

対象：

```text
店名
ジャンル
住所
電話番号
営業時間
定休日
HP有無
```

不要な要素の全探索を減らす。

---

### 7-2. 正規表現・全DOM検索の回数を減らす

現在、電話番号・営業時間・HP判定でページ全体のテキストを見る処理がある。

これらは便利だが重い可能性があるため、以下の順番にする。

```text
1. 明確なセレクタで取得
2. 取れなかった場合だけ全体テキスト検索
```

---

## 8. CSV出力の最適化

人気ジャンル一括取得では、ジャンルごとにCSVを出力している。

これは便利だが、ジャンル数が多いと処理が増える。

修正案：

```text
ジャンルごとCSV出力：ON / OFF
最後にまとめて出力：ON / OFF
```

推奨：

```text
通常運用：ジャンルごとCSV出力ON
高速運用：最後にまとめて出力ON
```

---

## 9. ログ表示の最適化

現在は進捗ログが細かく出るため、件数が多いとUI更新も負荷になる。

修正案：

```text
ログ更新は1件ごとではなく、5件ごとにまとめる
成功ログは最新店舗名だけ表示
詳細ログはエラー時だけ出す
```

# 修正内容サマリー（Claude Codeへの依頼文）

食べログスクレイピングの速度改善を行ってください。

現在の拡張機能は、食べログの詳細ページ取得時に待機時間を入れながら並列取得しています。初期設定は食べログの並列数2、待機2000msで、安全寄りですが遅いです。

以下を修正してください。

1. 食べログの並列数上限を3から5に変更する（`src/popup.js`、および `popup.html` の該当input要素の `max`）
2. 食べログの待機時間下限を1000msから500msに変更する（`src/popup.js`、および `popup.html` の該当input要素の `min`）
3. UIに安全モード・標準モード・高速モードを追加する
4. 標準モードは並列3、待機1000ms、リトライ1、タイムアウト15秒にする
5. 高速モードは並列4、待機700ms、リトライ0、タイムアウト10秒にする
6. 403 / 429 / ブロック画面検知時は、既存通り自動減速する（変更しない）
7. 人気ジャンル一括取得のジャンル間待機を2000msから500msに短縮する（`src/offscreen.js` 1762行目付近）
8. ただし、アクセス制限検知時はジャンル間待機を5000msにする
9. 人気ジャンル一括取得では、親タスク側に `globalSeenUrls` を持たせ、ジャンルをまたいだ同一店舗URLの再取得を防ぐ（`src/offscreen.js` 1696行目付近のループ）
10. 一覧ページ取得と詳細ページ取得を分離する
11. 先にページネーションで店舗URLだけを全件収集する
12. 収集した店舗URLを重複排除する
13. 重複排除後に、詳細ページを並列取得する
14. ログ更新は1件ごとではなく、5件ごとにまとめる
15. CSV出力はジャンルごと出力と最後にまとめて出力を選べるようにする

完了条件は、同じ条件で取得した時に、失敗率を大きく増やさずに取得時間が短縮されることです。

## 実装時の注意

- 既存のブロック検知・自動減速・失敗URLログ・カウンター表示（成功/失敗/リトライ中/ブロック/タイムアウト/店名なし/住所なし/電話なし/HTML不一致/その他）の仕組みは壊さないこと。
- `popup.js` と `offscreen.js` 間のメッセージング（`chrome.runtime.sendMessage` / `onMessage` のペイロード形状）を変更する場合は、両ファイルを整合させること。
- ホットペッパー側の設定・処理には触れない（今回は食べログのみが対象）。
- 関数名・変数名が本要件書のコード例と異なる場合は、既存コードのスタイル・命名規則に合わせて実装してよい。意図（速度向上・重複排除・フェーズ分離）を優先すること。
