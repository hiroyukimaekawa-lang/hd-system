# 電話番号補完・デュアルプロファイル運用 導入手順（Ver10.0.1）

対象: `gas/list-normalizer/code-restaurant.js`（要件書「HDリスト生成システム改修要件書」対応）

---

## 1. 変更概要

| 項目 | 内容 |
|---|---|
| プロファイル切替 | `SYSTEM_PROFILE`（ELECTRIC=電気営業用 / AFFILIATE=リアルアフィリエイト用）で、ビル・階数表記候補の除外有無を同一コードのまま切り替え |
| 共通除外 | 道の駅・カラオケ・総合公園・ゴルフ・ホテル・スーパー・イオンモールは両プロファイルで除外（既存の`FACILITY_EXCLUDE_KEYWORDS`も全維持、共通除外はプロファイルより優先） |
| 電話番号補完 | 電話番号未取得の営業候補をSerpAPI（google_maps検索→place_id詳細→通常Google検索の補助確認）で再検索し、信頼度スコアで自動補完（85点以上）/要確認（70〜84点）/見送りに分類 |
| セキュリティ | SerpAPIキーをコードから削除し、スクリプトプロパティ`SERPAPI_API_KEY`へ移行（`code.js`も同様） |
| CSV | 31列構成・列順は変更なし。ファイル名を`コムデスク_<運用区分>_<エリア>_<ジャンル>_<日付>.csv`へ変更 |

## 2. ELECTRICとAFFILIATEの動作差分

| 動作 | ELECTRIC（電気営業用） | AFFILIATE（リアルアフィリエイト用） |
|---|---|---|
| ビル・階数表記（ビル/テナント/プラザ/タワー/センター/B1F/1F〜5F/○階/地下） | 施設判定「確認対象」→ 02_確認対象へ。営業対象・コムデスクCSVに出力しない | この表記のみでは除外しない（施設判定「対象」、理由欄に表記を記録）。他条件を満たせば営業対象 |
| 共通除外7種＋既存大型施設除外 | 除外（共通） | 除外（共通・プロファイルより優先） |
| 電話番号補完の対象 | ビル・階数表記の確認候補はAPI検索対象外 | ビル・階数表記があっても他条件を満たせばAPI検索対象 |
| CSVファイル名の運用区分 | `電気営業` | `リアルアフィリエイト` |

## 3. 新環境の作成手順（環境ごとに1回）

1. スプレッドシートを新規作成する。
   - `HDリスト生成_電気営業用` / `HDリスト生成_リアルアフィリエイト用`
   - それぞれ別のDriveフォルダに置く（CSV投入フォルダ・処理済みフォルダ・完成版CSVエクスポートはスクリプトが親フォルダ配下へ自動作成するため、親フォルダを分ければ入出力は完全に分離される）。
2. 各シートに`MASTER_CHAIN`タブを既存環境からコピーする。
3. 拡張機能 > Apps Script を開き、`code-restaurant.js`の内容を貼り付けて保存する。
4. スクリプトプロパティを設定する（次章）。
5. 既存のスプレッドシート・フォルダは検証完了まで変更せず残す（バックアップ）。

## 4. スクリプトプロパティ設定手順

Apps Scriptエディタ > 左メニュー「プロジェクトの設定」>「スクリプト プロパティ」で登録する。

| キー | 値 | 備考 |
|---|---|---|
| `SERPAPI_API_KEY` | 新しく発行したAPIキー | **必ず新規発行したキーを使う**（旧キーはコード・Git履歴に露出しているため失効させること） |
| `SYSTEM_PROFILE` | `ELECTRIC` または `AFFILIATE` | 必須。未設定だと明確なエラーで停止 |
| `MAX_SERPAPI_CALLS_PER_RUN` | `100` | 1回の実行のAPI呼び出し上限（省略時100） |
| `MIN_AUTO_ACCEPT_SCORE` | `85` | 自動採用の最低スコア（省略時85） |
| `PHONE_ENRICHMENT_DRY_RUN` | `true` | 初回検証は必ず`true`。`false`と明記した場合のみ本反映 |

### SerpAPIキーのローテーション（必須）

1. <https://serpapi.com/manage-api-key> で旧キーを失効（またはローテーション）する。
2. 新キーを発行し、上記スクリプトプロパティにのみ保存する。
3. キーをGitHub・スプレッドシート・CSV・ログ・READMEへ書かない。

※ Git履歴の書き換えは今回のスコープ外。旧キーの失効により無効化する。

## 5. ドライランの実行手順

1. `PHONE_ENRICHMENT_DRY_RUN=true` を確認。
2. CSVを「CSV投入フォルダ」へ入れ、メニュー「⚙️ HD店舗管理システム > 🚀 すべての一括処理を実行」を実行
   （または単体で「📞 電話番号補完を実行（SerpAPI）」）。
3. ドライラン中に行われるのは、SerpAPI検索・候補保存・スコア算出・取得元/照合内容の保存・確認シートとログの作成のみ。
   元の電話番号・営業対象件数・コムデスクCSV・既存分類は一切変更されない
   （高信頼候補はステータス`高信頼候補(ドライラン)`として01_NORMALIZEDの管理列に記録される）。
4. 確認するもの:
   - `01_NORMALIZED`の管理列（電話番号候補/電話補完ステータス/信頼度/取得元URLほか）
   - `確認_電話番号候補`シート（70〜84点および自動採用禁止条件に該当した候補）
   - `電話番号補完ログ`シート（API種別・HTTP結果・判定理由）

## 6. 本反映の実行手順

1. 候補精度を確認後、`PHONE_ENRICHMENT_DRY_RUN=false`へ変更。
2. 「🚀 すべての一括処理を実行」を実行。§13の順序で処理される:
   CSV取込 → 正規化〜ワークフロー分類 → 補完対象抽出 → SerpAPI検索・信頼度判定 →
   高信頼のみ01_NORMALIZEDへ反映 → 再判定（反映0件なら省略） → タブ振り分け → 営業対象 → コムデスクCSV → サマリー → 補完結果表示。
3. 既存の電話番号が入っている行は上書きされない。補完により電話番号重複が発生した行は既存の重複ルールで除外される。

## 7. 中断と再開（Ver10.0.1）

- 安全停止時間は**一括処理全体（executeAllProcesses）の開始時点**から計測する（CSV取込・初回判定を含めて約4分で補完を安全停止）。API上限到達時・401/403認証エラー時も同様に停止する。
- **補完が途中停止した場合、営業対象シート生成・コムデスクCSV出力・件数サマリーは確定されない。** 画面には「途中停止である旨・処理済み件数・残件数・再開方法・完成版CSVが未出力である旨」が表示される。
- 再開方法: メニュー「📞 電話番号補完を実行（SerpAPI）」の再実行、または`resumePhoneEnrichmentFromTrigger`の時間主導トリガー。停止した行から再開する。
- **再開で補完が完了すると、CSVを再取込せず現在の01_NORMALIZEDを使って、正規化→重複→チェーン→施設→ワークフローの再判定→分類シート→営業対象→コムデスクCSV→件数サマリー→補完結果表示まで自動実行される。** 再び途中停止した場合はカーソルを保存して終了する。
- カーソルはプロファイル別（`PHONE_ENRICHMENT_CURSOR_ELECTRIC`等）。先頭からやり直す場合はメニュー「📞 電話補完の再開カーソルをリセット」。
- HTTP 429/5xxは指数バックオフ（1s→2s→4s）で最大3回再試行。**401/403は一括処理全体を停止**し、停止行のカーソルを保存する。スクリプトプロパティのAPIキー修正後に再実行すると停止行から再開できる。

## 8. 管理列・追加シート

- `01_NORMALIZED`管理列（コムデスクCSVには出力されない）:
  電話番号候補 / 電話補完ステータス（未検索・高信頼補完・高信頼候補(ドライラン)・要確認・見つからず・APIエラー・対象外）/
  電話補完信頼度 / 電話補完取得元 / 電話補完取得元URL / 電話補完照合店名 / 電話補完照合住所 /
  電話補完検索クエリ / 電話補完実行日時 / 電話補完API呼出回数 / 電話補完エラー / システム区分
- `確認_電話番号候補`: 自動採用できない候補の人手確認用（手動判定欄つき、追記式・重複追加なし）
- `電話番号補完ログ`: API実行と判定の監査ログ（追記式）

## 9. 信頼度スコアと自動採用条件

| 判定項目 | 最大点 |
|---|---:|
| 店舗名一致（完全40 / 部分包含30） | 40 |
| 都道府県・市区町村一致 | 20 |
| 町域・番地一致（各10） | 20 |
| 店舗ジャンル一致 | 10 |
| Google Maps ID・公式サイト裏付け | 10 |

85点以上→自動補完候補、70〜84点→要確認、69点以下→自動採用しない。
スコアに関係なく自動採用しない条件: 支店名不一致 / 市区町村・番地不一致 / 同名候補複数 /
1位2位のスコア差15点未満 / 電話番号競合 / 閉業・移転の可能性 / 店名のみ一致 / 単一第三者サイトのみ / 電話番号形式不正。

## 9.5 通常Google検索候補の照合（Ver10.0.1）

通常Google検索の候補は、検索結果のタイトル・スニペット（情報源テキスト）に
店舗名・住所（市区町村＋町域または番地）が**実際に含まれるかを検証**してからスコアリングする。
対象店舗の店名・住所を候補側へそのまま代入することはない。
自動採用できるのは「店舗名一致＋住所一致＋電話番号＋（公式サイト or 独立した2情報源）」を
情報源側で確認できた場合のみ。住所を情報源で確認できない候補は、電話番号が取得できていても
自動採用せず`要確認`（確認_電話番号候補シート）へ送られる。

## 11. 残課題と既知の制約

- **古河市データの改修前後件数比較**: 実スプレッドシートでの実行が必要なため未実施。改修前基準値（営業対象504 / 重複除外1,015 / チェーン除外62 / ビル管理除外7 / 電話番号なし確認95 / 小規模ビル確認29 / 住所確認2）に対し、ドライラン→本反映後に`00_件数サマリー`と各タブ件数で比較表を作成すること。
- 旧APIキーはGit履歴に残存している。履歴書き換えはスコープ外のため、**旧キーの失効が必須**。
- CSVを再取込すると`01_NORMALIZED`が作り直されるため、補完管理列は取込単位でリセットされる（同一取込内の再実行では冪等）。
- SerpAPIの応答フィールド（`local_results`/`place_results`/`knowledge_graph`）は実データで揺れがあるため、ドライランで取得元・照合結果を確認のうえ本反映すること。
- `code.js`（旧テンプレート）のSerpAPI補完は従来ロジックのまま（キー管理のみスクリプトプロパティ化）。新ロジックは`code-restaurant.js`のみ。
- ファイル名は「コムデスク_<運用区分>_<エリア>_<ジャンル>_<日付>.csv」（ジャンル別出力を維持したまま運用区分を追加。発注者確認済み）。
